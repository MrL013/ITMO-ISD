# Этап 3. Тестирование и анализ AgroVision

## 1) Что тестировалось
1. Функциональность API:
- получение списка полей;
- валидация телеметрии;
- генерация рекомендаций.

2. Корректность fallback-режима:
- если LLM/API недоступны, система формирует локальную рекомендацию.

3. Базовая производительность:
- серия параллельных запросов к `GET /api/fields`.

## 2) Команды тестирования

```bash
npm test
npm run bench
```

## 3) Фактические результаты (31 марта 2026)

### Автотесты API
- `3/3` теста прошли успешно.
- Общая длительность: `~0.87 c`.

Ключевые проверки:
- `GET /api/fields` возвращает предзаполненные поля.
- `POST /api/recommendation` возвращает рекомендацию и источник (`openai`/`fallback`).
- `POST /api/telemetry` корректно валидирует payload (`400` на невалидный, `201` на валидный).

### Производительность (benchmark)
Параметры: `200` запросов, конкурентность `20`.

Результат:
- `Avg latency: 30.26 ms`
- `P95 latency: 83.63 ms`
- `Throughput: 607.63 req/s`
- `Memory RSS: 56.96 MB`

## 4) Что выявило тестирование
1. Выявленная проблема:
- первично запрос рекомендаций занимал около `5 секунд` из-за ожидания внешнего LLM-вызова.

2. Исправление:
- добавлен таймаут `OPENAI_TIMEOUT_MS` (по умолчанию `1500 ms`) и мгновенный fallback к локальной эвристике.

3. Эффект:
- тест сценария рекомендаций ускорился примерно до `~0.49 c`.

## 5) Степень успеха разработки
- Прототип полностью работоспособен локально (UI + API + рекомендации).
- Этапы 1-2-3 закрыты: проектирование, реализация, тестирование/анализ.
- Ограничения соответствуют уровню прототипа (без production-инфраструктуры и полноценной GIS-визуализации).

## 6) Возможности развертывания
1. Локально/демо: один `Node.js` процесс.
2. Сервер/VM: `pm2` + reverse proxy (`Nginx`).
3. Контейнер: Docker-образ и деплой в Kubernetes.
4. PaaS: Render/Railway/Fly.io для пилота.

Для production рекомендуются:
- вынесение БД и брокера сообщений;
- централизованный логинг/метрики;
- rate limit, авторизация и секрет-менеджмент для API-ключей.
